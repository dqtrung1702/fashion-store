Tài liệu dành cho lập trình viên tiếp quản dự án. Nội dung mô tả stack, module chính, API, vận hành local và các điểm còn thiếu.

2. Cách chạy local

cd /home/fashion-store
cp backend/.env.example backend/.env
./start.sh

Trang bán hàng:  http://localhost:5173
API dịch vụ:     http://localhost:8000
Swagger:         http://localhost:8000/docs
MinIO Console:   http://localhost:9001
Mặc định ./start.sh chạy bằng image Docker local với docker compose up -d --no-build, nên có thể chạy offline sau khi image đã được pull/build một lần. Không dùng --build khi đang mất mạng vì Docker sẽ kiểm tra metadata của base image trên Docker Hub.

Frontend local hiện tại

  • Compose dev hiện chạy npm ci --include=optional && npm run build && npm run preview -- --host 0.0.0.0.
  • Cách này phục vụ bundle production ở cổng 5173, không inject /@vite/client hoặc React Refresh, nên không cần unsafe-eval trong CSP.
  • Khi sửa source frontend, cần rebuild/restart frontend container để thấy thay đổi. Dữ liệu admin/content/product vẫn cập nhật động qua API.

Deploy frontend lên MinIO legacy/tuỳ chọn

  • Chạy ./scripts/deploy_frontend_to_minio.sh để build Vite với base path theo bucket và upload frontend/dist lên MinIO.
  • Bucket mặc định: fashion-store-frontend.
  • URL local sau deploy: http://localhost:9000/fashion-store-frontend/index.html.
  • Có thể đổi bucket bằng FRONTEND_BUCKET=my-frontend ./scripts/deploy_frontend_to_minio.sh.
  • MinIO serve được static file, nhưng refresh trực tiếp các route SPA như /collections/... vẫn cần reverse proxy fallback về index.html.

Chạy Docker offline

  • Build/pull lần đầu khi có mạng: ./start.sh --build.
  • Chạy hằng ngày hoặc khi offline: ./start.sh.
  • Image local cần có: mongo:6.0, minio/minio:latest, redis:7-alpine, fashion-store-backend:latest, fashion-store-frontend:latest.
  • Đóng gói image để chuyển máy: docker save mongo:6.0 minio/minio:latest redis:7-alpine fashion-store-backend:latest fashion-store-frontend:latest -o fashion-store-images.tar.
  • Nạp image trên máy offline: docker load -i fashion-store-images.tar.
Tài khoản admin local mặc định: admin / Admin123!. Khi ADMIN_SEED_ENABLED=true, dịch vụ API đảm bảo tài khoản này tồn tại lúc khởi động.

2.1. Cấu hình production Nginx

Production có bộ file riêng để chạy Nginx làm entrypoint duy nhất:

cp .env.production.example .env.production
docker compose --env-file .env.production -f docker-compose.prod.yml up -d --build
  • nginx/Dockerfile build frontend rồi copy dist vào image nginx:1.27-alpine.
  • nginx/default.conf serve SPA, proxy /api/ sang backend và proxy /fashion-store-assets/ sang MinIO.
  • docker-compose.prod.yml chỉ public Nginx ra HTTP_PORT; MongoDB, Redis, MinIO và backend nằm trong Docker network.
  • .env.production.example là mẫu env cần đổi secret/password/domain trước khi dùng.
  • CORS production đọc từ CORS_ORIGINS. URL ảnh public đọc từ MINIO_PUBLIC_ENDPOINT và MINIO_PUBLIC_SECURE.

2.2. Dữ liệu runtime, backup và snapshot Git

Sở hữu dữ liệu runtime

  • MongoDB lưu users, products, collections, site content, orders, authenticated carts và authenticated wishlists.
  • MinIO lưu ảnh sản phẩm public, hiện dùng bucket mặc định fashion-store-assets.
  • Redis chỉ cache public catalog/content; không phải nguồn dữ liệu gốc.
  • localStorage vẫn được dùng cho auth cache, guest cart và guest wishlist.

Xuất snapshot để commit lên Git

Không commit Docker volumes trực tiếp. Xuất chúng ra file thường trước:

./scripts/export_runtime_data.sh

git add data-snapshots/current
git commit -m "Add runtime data snapshot"
git push

Có thể dùng thư mục theo ngày:

./scripts/export_runtime_data.sh "data-snapshots/$(date +%F)"

Backup production

MongoDB có thể archive từ container production:

mkdir -p backups
docker compose --env-file .env.production -f docker-compose.prod.yml exec mongodb \
  sh -c 'mongodump --username "$MONGO_INITDB_ROOT_USERNAME" --password "$MONGO_INITDB_ROOT_PASSWORD" --authenticationDatabase admin --archive=/tmp/fashion-store.gz --gzip'
docker compose --env-file .env.production -f docker-compose.prod.yml cp mongodb:/tmp/fashion-store.gz ./backups/fashion-store-$(date +%F).gz

MinIO production hiện nằm trong Docker volume minio_data; muốn backup đúng thì dùng volume backup, object replication hoặc chuyển media sang S3/R2/CDN-managed storage.

Review cookbook: điểm vào nhanh

  • Routing: frontend/src/App.jsx, frontend/src/routes/publicRoutes.js, frontend/src/routes/pageRegistry.js, frontend/src/routes/AdminRoute.jsx.
  • Admin UI shell: frontend/src/pages/AdminPage.jsx.
  • Admin helper/transform: frontend/src/features/admin/constants.js, transformers.js, components.jsx, viewUtils.js.
  • Catalog state: frontend/src/store/catalogStore.js, frontend/src/lib/catalog.js, frontend/src/lib/text.js.
  • Content state: frontend/src/store/contentStore.js.
  • Backend entry: backend/main.py, backend/routes/*.

Review cookbook: theo loại thay đổi

  • Nếu sửa route hoặc thêm page mới, bắt đầu ở App.jsx và thư mục frontend/src/routes/.
  • Nếu sửa admin form/import/export, mở frontend/src/features/admin/transformers.js trước, chỉ quay lại AdminPage.jsx khi cần xem wiring UI.
  • Nếu sửa UI field/button dùng lại nhiều nơi trong admin, đọc frontend/src/features/admin/components.jsx.
  • Nếu review lỗi parse slug/list/structured text, đọc frontend/src/lib/text.js trước rồi mới tới store hoặc admin transformer.
  • Nếu sửa catalog CRUD, tập trung vào frontend/src/store/catalogStore.js và frontend/src/lib/catalog.js.
  • Nếu sửa content localization hoặc sync backend, đọc frontend/src/store/contentStore.js.
Hotspots hiện tại: frontend/src/pages/AdminPage.jsx vẫn là file lớn nhất; frontend/src/store/catalogStore.js, frontend/src/store/contentStore.js và backend/routes/admin.py vẫn là các điểm nên mở sớm khi review.

5. Hành vi admin hiện tại

  • Màn hình admin mặc định mở ở Marketing monitor để kiểm tra sức khỏe landing page, CTA, campaign, social proof, UGC và nguồn sản phẩm hỗ trợ chiến dịch.
  • Homepage không còn phụ thuộc hoàn toàn vào các heading cố định trong code; tab nội dung có phần Homepage blocks để đổi eyebrow/title/CTA và bật tắt từng block bằng dòng key | eyebrow | title | CTA label | CTA link | on/off.
  • Nhóm menu Shop, category tile trên homepage, các card phụ đầu trang, block theo dịp, nhóm danh mục ở footer và trang collection public lấy từ các collection active trong Admin. Không còn trộn các link editorial cố định như Tết & Hội Xuân vào navigation public; nếu CTA homepage trỏ tới collection slug không còn active, storefront tự chuyển về collection active đầu tiên.
  • Website public hỗ trợ hai ngôn ngữ vi và en. Content API lưu được locales.vi/locales.en; tab nội dung trong Admin có bộ chọn VI/EN để sửa từng ngôn ngữ.
  • Form sản phẩm và danh mục có vùng Bản dịch tiếng Anh. Storefront ưu tiên translations.en khi khách chọn EN, field nào để trống sẽ fallback về nội dung gốc.
  • Một sản phẩm có thể thuộc nhiều danh mục qua collectionSlugs. Backend vẫn giữ collectionSlug/category làm trường chính để tương thích, nhưng lọc public kiểm tra cả danh sách nhiều danh mục.
  • Admin có template Excel, import hàng loạt sản phẩm, export sản phẩm hiện tại ra Excel và bỏ qua ảnh. Ảnh có thể bổ sung sau qua kho MinIO/admin form.
  • Font public dùng cặp local UVNMuaThu.ttf + Nunito qua @font-face trong frontend/src/index.css: UVN Mua Thu cho logo, .font-display, hero title, campaign title và section title; Nunito cho body, menu, button, form và admin. Các file được lưu trong frontend/public/fonts để không phụ thuộc dịch vụ font bên ngoài lúc runtime.
  • Bộ font active hiện gồm UVNMuaThu.ttf, Nunito-400.ttf, Nunito-500.ttf, Nunito-600.ttf, Nunito-700.ttf. Các font local cũ như Lora, VLSelfie-Regular.otf, NVNAzkia.ttf và 1FTVBillionDreams.ttf chỉ được giữ lại làm phương án cũ, không còn được stylesheet active tham chiếu.
  • Nguồn font hiện hành: Nunito được lấy từ Google Fonts; VLSelfie-Regular.otf từng lấy từ Font2s với ghi chú Free for Personal Use, nên phải giữ ở trạng thái inactive trừ khi xác minh lại license cho production.
  • Typography scale đã được hạ bớt sau khi đổi display font: hero/page title dùng mức nhỏ hơn, card title giảm từ các mức quá lớn, và text-wrap: balance được áp cho heading để hạn chế dòng cuối chỉ còn 2-3 chữ. Màu chữ trong frontend/src/index.css được remap sang tông xanh lá trầm + accent coral/rose để ăn với nền pastel hiện tại.
  • Thông số upload ảnh chi tiết đã được nhúng trực tiếp trong docs/admin-handoff-guide.html, gồm tỉ lệ, kích thước khuyến nghị, kích thước tối thiểu và lưu ý crop cho từng khung hero/product/campaign/UGC.
  • Hướng dẫn vận hành chi tiết trang /admin nằm ở docs/admin-handoff-guide.html, bao gồm quyền truy cập, từng tab, format nhập liệu, API, dữ liệu bị ảnh hưởng và lỗi thường gặp.
  • Định hướng storefront hiện tại là image-led marketing: homepage, campaign, collection và merchandising hero ưu tiên ảnh lớn, tile ảnh overlay chữ ngắn; tránh thêm nhiều đoạn mô tả dài ở trang public.
  • Đã gỡ các nút shortcut trùng lặp ở đầu trang.
  • Đã gỡ các khối trạng thái dữ liệu kỹ thuật khỏi UI.
  • Đã gỡ các nút reset dữ liệu mẫu khỏi UI.
  • Kho ảnh có thể tải lên, liệt kê, copy URL, chọn cho sản phẩm và xóa.
  • Không xóa được ảnh nếu có sản phẩm đang dùng.
  • Public products/collections/categories/content được cache bằng Redis. Các thao tác admin mutate catalog/content sẽ tự xóa cache tương ứng.
  • Token hết hạn hoặc sai sẽ chuyển người dùng về trang đăng nhập.

Troubleshooting local Docker

  • Dừng stack local: docker compose down. Nếu cần xóa luôn volumes: docker compose down -v.
  • Khi service lỗi khởi động, reset và build lại: docker compose down -v, docker compose build --no-cache, docker compose up -d.
  • Xem log nhanh: docker compose logs -f backend, frontend, mongodb, minio, redis.
  • Nếu port 5173, 8000, 9000 hoặc 9001 bị chiếm, dừng service khác hoặc đổi mapping trong docker-compose.yml.

Cookbook vận hành nhanh

  • Frontend local dùng production build preview; khi sửa source frontend bằng Docker compose cần rebuild hoặc restart frontend container.
  • Upload ảnh sản phẩm có thể test ngoài UI bằng POST /api/admin/uploads/products với multipart file và bearer token admin.
  • Redis chỉ là cache mềm; nếu Redis hỏng, ứng dụng vẫn phải phục vụ bằng MongoDB và tự bỏ qua cache qua redis_client.py.
  • Tài liệu cập nhật nhất để handoff kỹ thuật toàn hệ thống là file HTML này, cộng với docs/data-model.html và docs/architech.html.

8. Khoảng trống kỹ thuật

Chưa có HTTPS tự động trong compose production, cổng thanh toán, nhật ký thao tác admin, phân quyền chi tiết, soft delete, tự merge giỏ hàng guest khi đăng nhập và chính sách media private/CDN đầy đủ.
